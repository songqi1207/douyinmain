# 三类短视频工作流生成降本与去平台依赖方案

> 版本：2026-07-15 讨论稿
>
> 范围：书单、香烟、神话三类工作流
>
> 本文成本以“生成一条可进入剪映草稿的视频素材”为统计口径，不含人工审核、视频发布和投流费用。

## 一、结论摘要

建议采用“两步走”方案：

1. **短期先降图片成本**：保留现有扣子工作流，把批量生图从米核即梦切换到 Google Gemini 2.5 Flash Image Batch；计划任务走 Batch，紧急任务保留实时通道。
2. **中期摆脱扣子编排**：由本地 Python 直接完成文案、分镜、配音、时间线和 `draft_key` 生成，再写入本机剪映草稿；生图继续通过可插拔接口调用 Google Batch，未来达到足够规模后再评估本地 GPU 生图。

按现有图片数量和预算单价计算：

- 米核即梦：约 **¥0.20/张**。
- Google Batch：官方价 **$0.0195/张**；按 2026-07-14 美元兑人民币中间价 6.799 折算约 **¥0.1326/张**，方案预算按 **¥0.13/张**计算。
- 单看图片生成，Google Batch 比米核约节省 **35%**。
- 三类工作流各生成一条，共 27 张图，图片成本由 **¥5.40 降至 ¥3.51**，节省 **¥1.89**。
- 根据模板预测，扣子非图片运行成本建议暂按 **¥0.55～¥0.60/条**预留；这是预算值，不是账单实测值。

本文以图片生成成本为主要比较对象，并根据当前三个工作流模板对扣子非图片运行成本做预算级预测。预测用于会议预算，不等同于扣子账单实测；服务托管、失败重试和人工审核仍不计入本轮对比。

### 2.4 根据当前模板预测扣子单次运行成本

当前模板均使用 `豆包·1.5·Pro·32k`。扣子官方价格为输入 **¥0.0008/千 Token**、输出 **¥0.002/千 Token**。配音按当前默认的大模型系统音色估算，采用火山引擎语音合成大模型刊例价 **¥45/10万字**，即约 **¥0.00045/字**。

口径提醒：本节为了与前面的图片成本表一致，继续按书单 10 张、香烟 8 个分镜、神话 9 个分镜测算。母版默认值并不完全相同：书单母版 `img_count` 为 6，但当前网页默认会覆盖为 10；新版香烟和神话母版的 `shuliang` 默认均为 14。每多一个香烟或神话分镜，会增加 1 次模型、1 次配音、2 次字幕对齐和约 1 次生图，必须按实际输入数量重新计算。

此外，当前网页 `/generate_cigarette` 接口仍使用旧香烟管线，而不是根目录的 `烟工作流模板_香烟鉴赏-v1.json`。因此下表的香烟预测对应新版模板；若会议讨论的是当前网页线上管线，应单独以旧模板运行记录为准。

模板调用结构如下：

| 工作流 | 模型调用结构 | 配音结构 | 其他工具结构 |
|---|---|---|---|
| 书单 | 2 次：口播稿、整批配图提示词 | 3 组批量节点，合计约 230～300 字 | 约 44 个非生图工具节点 |
| 香烟 | 2 次外层调用 + 每个分镜 1 次；按 8 个分镜约 10 次 | 开场、标题及 8 段正文 | 约 23 个外层工具节点，并有 16 次字幕对齐 |
| 神话 | 2 次外层调用 + 每个分镜 1 次；按 9 个分镜约 11 次 | 开场、标题及 9 段正文 | 约 23 个外层工具节点，并有 18 次字幕对齐 |

其中香烟和神话的“自动生成文案”与“参考文案改写”是二选一分支，不会同时收费。分镜循环中的模型、配音和字幕对齐会随分镜数量线性增加。

按模板提示词长度、目标输出字数和默认音色估算：

| 工作流 | 模型 + 配音预测 | 工具调用预留 | 建议预算值/条 |
|---|---:|---:|---:|
| 书单 | ¥0.12～¥0.16 | ¥0.35～¥0.45 | **¥0.60** |
| 香烟 | ¥0.10～¥0.15 | ¥0.30～¥0.40 | **¥0.55** |
| 神话 | ¥0.11～¥0.17 | ¥0.30～¥0.40 | **¥0.60** |

“工具调用预留”不是扣子已确认账单，而是考虑到模板内剪映草稿、字幕对齐、时间线、音频和图片装配等大量插件节点，以“可收费工具平均约 ¥0.01/次”的压力测试口径做的预算缓冲，并未认定这些插件实际都按此价格收费。若这些工具实际免费，单条扣子成本可能更接近“模型 + 配音预测”；若某个插件单独收费，则以插件页面和实际账单为准。

将上述预算值加入图片成本后，可得到便于会议讨论的单条总预算：

| 工作流 | 扣子 + 米核 | 扣子 + Google Batch | 单条节省 |
|---|---:|---:|---:|
| 书单 | 约 ¥2.60 | 约 ¥1.90 | ¥0.70 |
| 香烟 | 约 ¥2.15 | 约 ¥1.59 | ¥0.56 |
| 神话 | 约 ¥2.40 | 约 ¥1.77 | ¥0.63 |

这说明短期换 Google 主要节省图片费用；中期移除扣子后，理论上还可再释放约 **¥0.55～¥0.60/条**的预算，但本地编排仍可能需要另付大模型和配音 API 费用，不能直接视为全部净节省。

## 三、四种实施方案对比

| 方案 | 图片成本 | 编排方式 | 一次性开发量 | 交付速度 | 平台依赖 | 建议 |
|---|---:|---|---:|---|---|---|
| A. 现状：扣子 + 米核 | ¥1.60～¥2.00/条 | 扣子 | 无 | 实时 | 高 | 仅作为基线和紧急回退 |
| B. 扣子 + Google Batch | ¥1.04～¥1.30/条 | 扣子 | 低 | 最长约 24 小时 | 中高 | **短期推荐**，最快实现图片降本 |
| C. 本地 Python + Google Batch | ¥1.04～¥1.30/条 | 本地 Python | 中 | 最长约 24 小时 | 低 | **中期目标**，兼顾成本与稳定性 |
| D. 全本地模型 + 本地剪映 | API 边际成本接近 0 | 本地 Python | 高 | 取决于 GPU | 最低 | 高产量且有运维能力后再评估 |

### 为什么不直接全本地

本地生图虽然没有按张 API 费，但有显卡折旧、电费、模型维护、队列管理和人工运维。它是否更便宜取决于月图片量：

```text
本地单张成本 = 每月固定成本 ÷ 每月可用图片数
与 Google Batch 的盈亏平衡图片数 = 每月固定成本 ÷ ¥0.13
```

例如本地设备折旧、电费和维护合计按 ¥1,500/月估算，需要超过约 **11,539 张可用图片/月**，才可能在纯成本上低于 Google Batch；若固定成本为 ¥3,000/月，盈亏平衡约为 **23,077 张/月**。这只是决策示例，不是硬件报价，还必须考虑画质和可用率。

## 四、推荐落地方案

### 阶段一：Google Batch 小规模 A/B 验证

每类工作流选择 20 个真实主题，同一提示词分别调用米核和 Google Batch：

- 共测试 60 条视频、540 张图/供应商。
- 米核测试图片预算约 ¥108。
- Google Batch 测试图片预算约 ¥70.20。
- 两边合计测试预算约 ¥178.20，不含重试。

必须记录以下指标：

1. 首次生成可用率。
2. 人物、书封、烟盒主体一致性。
3. 提示词遵循度和中文文字错误率。
4. 平均与 P95 出图时间。
5. 平均重试次数。
6. 每张最终可用图片的真实成本。

如果 Google 标价低但可用率明显更差，实际节省会被重试抵消。因此最终比较指标应为：

```text
每张可用图片成本 = 实际总账单 ÷ 审核通过图片数
```

### 阶段二：形成混合生产通道

- 日常排产、夜间批量任务：Google Batch。
- 当日紧急任务或 Batch 超时：保留米核/实时生图作为回退。
- 每个任务记录供应商、单价、耗时、重试和审核结果。
- 生图接口做成 Provider，可通过配置切换，不把 Google 或米核写死在工作流里。

### 阶段三：逐步移除扣子

当前项目已经具备本地剪映草稿写入和 `draft_key` 导入能力。后续由 Python 本地流水线完成：

```text
主题输入
→ 文案与分镜
→ 批量生图和配音
→ 音频时长探测
→ 时间线、字幕、运镜、特效
→ draft_key
→ 本机剪映草稿
```

迁移顺序建议：神话 → 香烟 → 书单。每迁移一类，先与扣子版本并行生成并验收，再切断该类工作流对扣子的依赖。

### 阶段四：达到规模后评估本地模型

当月图片量稳定超过盈亏平衡点，并且已有人员负责模型和 GPU 运维时，再对本地 ComfyUI 生图、CosyVoice 配音和本地 LLM 做正式评估。否则优先选择“本地编排 + Google Batch”，可以去掉扣子运行依赖，同时避免过早承担 GPU 固定成本。

## 五、建议决策

建议本次会议确认以下事项：

1. 批准约 ¥180 的双供应商 A/B 测试预算。
2. 确认 Google Batch 为计划任务首选，米核暂时作为紧急回退。
3. 启动“本地 Python 生成 `draft_key`”改造，目标是逐步移除扣子运行依赖。
4. 在月图片量和可用率数据稳定前，暂不采购专用 GPU。

## 六、当前工作流发布链路

当前页面默认使用 `WORKFLOW_BUILD_MODE=template`，先完成工作流母版验证：

```text
详情页输入主题
→ 后台复制对应工作流母版并写入主题默认值
→ 下载 Coze 工作流 JSON
→ 人工导入 Coze 逐个调试
→ 调试成功后发布并记录 workflow_id
```

全部发布后将 `WORKFLOW_BUILD_MODE` 切换为 `published`、`WORKFLOW_PROVIDER_MODE` 切换为 `coze`。后台通过统一执行器调用已发布工作流，不在前端暴露 Token：

```text
主题输入
→ Python requests / SDK 调用已发布 workflow_id
→ 获取 draft_key 或草稿 JSON
→ Windows 本地渲染 worker
→ MP4 地址
→ 前端直接播放
```

书单、香烟和神话仍使用仓库内自有模板生成器；参考工作流在发布前只生成可导入 JSON，避免把“母版生成成功”误报成“视频已经生成”。

## 七、会员注册审核链路

当前会员注册采用审批制，不再由访客自行设置密码并立即创建账号：

1. 访客在 `/business/register` 只提交邮箱，申请以 `pending` 状态写入 SQLite。
2. 管理员登录 `/business/admin/registrations` 查看待审核申请。
3. 管理员点击通过后，后台生成 12 位随机登录密码，并通过 SMTP 发到申请邮箱。
4. SMTP 返回发送成功后才激活账号；发信失败则账号保持不可登录，申请回到待审核并保留失败原因以便重试。
5. 用户使用“邮箱 + 邮件中的密码”登录。旧版已有用户名账号仍可继续用用户名登录。

管理员与邮件服务通过 `.env` 的 `SITE_ADMIN_EMAIL`、`SITE_ADMIN_PASSWORD`、`SMTP_HOST`、`SMTP_PORT`、`SMTP_USERNAME`、`SMTP_PASSWORD`、`SMTP_FROM` 配置。缺少 SMTP 时审核接口返回 `503`，不伪造已发信状态。
